Design Notes — Rappels Email des Réservations

Date : 20 juin 2026

Contexte et objectif

Cette note de design accompagne le PRD des rappels email de réservation.

Documents de référence :

  • PRD Réservations d'aéronefs
  • PRD Rappels email des réservations

L'objectif est de définir une architecture claire pour produire des rappels email fiables, sans coupler la logique métier de réservation avec la mécanique d'envoi.

Principes d'architecture

  • Séparer la source de vérité des réservations et la logique de notification.
  • Rendre le déclenchement de rappels configurable par section.
  • Garantir l'idempotence des envois pour éviter les doublons.
  • Centraliser la traçabilité des envois (succès/échecs) dans un journal exploitable.

Vue d'ensemble

Composants logiques

Composant Rôle
Reservation Source Fournit l'état des réservations actives et leurs transitions (création, modification, annulation).
Reminder Policy Calcule les échéances applicables à une réservation selon la configuration de section.
Reminder Scheduler Sélectionne les notifications à émettre à l'instant T.
Notification Composer Construit le contenu des messages (rappel, mise à jour, annulation).
Delivery Gateway Transmet les emails au service d'envoi configuré.
Delivery Log Conserve la trace des tentatives d'envoi et des résultats.
Admin Configuration Porte les paramètres de section : activation, échéances, règles de base.

Flux conceptuels

1) Planification des rappels

  1. Une réservation active est observée par le moteur de rappel.
  2. La politique de rappel applique les règles de section.
  3. Les échéances à venir sont matérialisées pour émission.

2) Emission des notifications

  1. Le scheduler récupère les notifications dues.
  2. Le compositeur génère le message selon le type d'événement.
  3. Le gateway envoie l'email.
  4. Le résultat est journalisé.

3) Gestion des changements

  1. Une modification significative de réservation déclenche une notification de mise à jour.
  2. Une annulation invalide les rappels futurs et déclenche une notification d'annulation.

Contrats fonctionnels

Contrat "éligibilité rappel"

Entrée : réservation + configuration section + horodatage courant.

Sortie :

  • éligible ou non,
  • type de notification,
  • liste de destinataires valides,
  • échéance associée.

Contrat "traçabilité"

Chaque tentative conserve au minimum :

  • identifiant de réservation,
  • type de notification,
  • destinataires,
  • date/heure d'émission,
  • statut (succès/échec),
  • message d'erreur éventuel.

Séparation des responsabilités

  • Le module réservation reste responsable de l'état métier des réservations.
  • Le module rappels reste responsable du calcul et de l'envoi des notifications.
  • Le module administration reste responsable des paramètres de section.
  • Le journal d'envoi reste la source d'observabilité pour le support et l'exploitation.

Risques et garde-fous

Risque Impact Garde-fou d'architecture
Doublon d'envoi Saturation utilisateur Idempotence sur le triplet (réservation, échéance, type).
Annulation non propagée Rappel erroné Invalidation explicite des rappels planifiés sur annulation.
Adresse email absente/invalide Non-réception Validation destinataires avant émission + journal d'échec.
Couplage fort au module réservation Maintenance difficile Contrat d'entrée/sortie stable entre source réservation et moteur de rappel.
